
> Extend `CMakeLists.txt` with a build-and-run target
>
> 1. Define a `function(add_runnable name)` which calls `add_executable`
> 2. Change `add_executable(hist1 hist1.cpp)` to `add_runnable(hist1)`.
> 3. Use `add_custom_target` to add a `run_${name}` target.
> 4. The path to your binary is `$<TARGET_FILE:${name}>`, a "generator expression".
> 5. `cmake --build build -- run_hist1` builds and executes in one step.
{:.block-task}

> Documentation

>[:green_book: `function`](https://cmake.org/cmake/help/latest/command/function.html),

>[:green_book: `$<TARGET_FILE:...>`](https://cmake.org/cmake/help/latest/manual/cmake-generator-expressions.7.html#genex:TARGET_FILE).

## Performance

> Let's consider performance.
>
> 1. Build with `-v` to see the call to `g++`.
>    Inspect the compiler flags.
> 2. `ls -l build/hist1`: Note down the binary size.
> 3. Run `hyperfine build/hist1` (if the time is below 300ms, increase `N`)
> 4. Run `perf stat build/hist1` (prepend `taskset 1` to inhibit cpu-migrations)
{:.block-warning}

> Configure in "Release" mode
>
> 1. CMake defaults to an empty `CMAKE_BUILD_TYPE`
> 2. use `ccmake` on the `build` directory to set it to `Release`
> 3. Redo `hyperfine` and `perf stat`
{:.block-tip}

> CMake defaults
>
> I disagree with the CMake build type default. Whenever you get a package and just build it with defaults you 
typically get an unoptimized Debug build.
> IMHO, the default should be `Release` or `RelWithDebInfo`.
>
> Whenever you change the build type, CMake ensures everything get's rebuild accordingly.
>
> Experiment with [`-fhardened`][4] and try to make an out-of-bounds access to trigger it.
{:.block-warning}

> Use the right tools
>
> - Did you use a `std::map`, as in the cppreference example? Consider a `std::vector<int>`.
> - Try with `std::list` and see what `perf stat` tells you.
{:.block-tip}
